# الخطة الرئيسية للمشروع (Project Master Plan)

## 1. رؤية المشروع (Project Vision)
**العربية**: إنشاء بوابة رسائل تعتمد على نموذج "جهازك هو البوابة" (BYOD)، مما يتيح للشركات استخدام أجهزة أندرويد الخاصة بها كبوابات لإرسال الرسائل النصية والواتساب. يوفر الحل طبقة إدارة مركزية (سيرفر آيديا) للاشتراكات والفوترة، بينما تتم عملية إرسال الرسائل فعلياً بشكل مباشر بين نظام العميل وجهاز الأندرويد عبر اتصال Socket.io لامركزي.

**English**: To create a "Bring Your Own Device" (BYOD) messaging gateway that allows businesses to use their own Android devices as SMS and WhatsApp gateways. The solution provides a management layer (Idea Backend) for subscriptions and billing, while the actual message execution happens directly between the client's system and the Android device via a decentralized Socket.io connection.

## 2. تحليل الفجوات (Gap Analysis)

### أ. فجوات تقنية حرجة (Critical Technical Gaps)

| الفجوة | الخطورة | الوصف | الإجراء المطلوب |
| :--- | :--- | :--- | :--- |
| **هشاشة أتمتة واتساب** | 🔴 عالية | يعتمد `WhatsAppService` على حقن الكود (DOM injection) باستخدام محددات ثابتة (مثل `aria-label="اكتب رسالة"`). هذا معرض للتعطل إذا قامت واتساب بتحديث واجهتها أو تغيرت لغة الجهاز. | تنفيذ استراتيجية محددات قوية (Robust Selectors) مع بدائل متعددة ومحددات لا تعتمد على اللغة. النظر في استخدام "خدمات الوصول" (Accessibility Services) لاستقرار أفضل. |
| **غياب تخزين طابور الرسائل** | 🔴 عالية | طابور الرسائل في تطبيق الأندرويد موجود في الذاكرة المؤقتة (RAM) فقط. إذا تعطل التطبيق أو أعيد تشغيله، تضيع الرسائل المعلقة. | تنفيذ تخزين دائم (SQLite) لطابور الرسائل داخل تطبيق الأندرويد. |
| **غياب فرض حدود الاستخدام** | 🟠 متوسطة | رغم جمع إحصائيات الاستخدام، لا توجد **آلية لحظية** لمنع الإرسال إذا تجاوز المستخدم حد الباقة، لأن الإرسال يتم مباشرة بين العميل والتطبيق. | يجب أن يقوم التطبيق بجلب "الرصيد المتبقي" من السيرفر بشكل دوري وفرض المنع محلياً قبل معالجة أوامر `send_message`. |
| **غياب أدوات الربط للعميل** | 🟠 متوسطة | يتعين على العملاء بناء سيرفر Socket.io الخاص بهم يدوياً. | إنشاء حزمة برمجية (SDK) بسيطة أو أمثلة كود (Node.js/PHP) لمساعدة العملاء على إعداد السيرفر بسرعة. |

## 3. خارطة الطريق (Roadmap)

### المرحلة 1: الاستقرار وحفظ البيانات (تقدير: أسبوعين)
*   [ ] **إصلاح**: تنفيذ طابور رسائل دائم في التطبيق (SQLite).
*   [ ] **إصلاح**: تحسين محددات واتساب (دعم الإنجليزية/العربية + محددات عامة).
*   [ ] **ميزة**: إضافة فرض الحصة (Quota Enforcement) محلياً في التطبيق.

### المرحلة 2: تجربة العميل (تقدير: أسبوع واحد)
*   [ ] **توثيق**: إنشاء "دليل إعداد سيرفر العميل".
*   [ ] **أدوات**: إصدار حاوية Docker أو حزمة Node.js لسيرفر العميل لتسهيل الإعداد.

### المرحلة 3: ميزات متقدمة (تقدير: 3 أسابيع)
*   [ ] **ميزة**: دعم المرفقات (صور/PDF) في واتساب.
*   [ ] **ميزة**: "وضع الحملات" في التطبيق للإرسال الجماعي مع فواصل زمنية.
*   [ ] **مراقبة**: إضافة "نبض" (Heartbeat) للسيرفر لتنبيه المستخدمين إذا انقطع اتصال جهاز البوابة الخاص بهم.
